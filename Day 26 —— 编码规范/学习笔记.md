本节目标：**建立可长期遵循的 C++ 编码规范认知**。这一节不讲语法，而讲**如何写出可维护、可协作、可扩展的代码**。

---

## 一、为什么编码规范如此重要？

* 代码是写给 **人** 看的，其次才是机器
* 单人项目 ≠ 工程代码
* 规范 = 降低理解成本 + 降低维护风险

> **技术债的本质，往往不是算法，而是混乱的结构与命名。**

---

## 二、header / source 分离

### 1. header / source 的职责划分

| 文件          | 职责     |
| ----------- | ------ |
| `.h / .hpp` | 声明（接口） |
| `.cpp`      | 定义（实现） |

原则：

> **头文件只描述“能做什么”，源文件才关心“怎么做”。**

---

### 2. 为什么要分离？

* 减少编译依赖
* 隔离实现细节
* 提升可维护性
* 支持多人协作

---

### 3. 头文件中应该放什么？

✅ 应该放：

* 类声明
* 函数声明
* enum / enum class
* 常量声明（constexpr / const）

❌ 不应该放：

* using namespace std;
* 函数实现（特殊 inline 除外）

---

### 4. include 守卫

```cpp
#ifndef MY_CLASS_H
#define MY_CLASS_H

// declarations

#endif
```

或：

```cpp
#pragma once
```

---

### 5. 前向声明（forward declaration）

```cpp
class Foo; // 前向声明
```

作用：

* 降低头文件依赖
* 加快编译速度

---

## 三、命名规范（工程重点）

### 1. 为什么命名比实现更重要？

* 好命名 ≈ 自文档代码
* 坏命名 ≈ 永久技术债

---

### 2. 常见命名风格

| 对象      | 推荐风格                   |
| ------- | ---------------------- |
| 类型 / 类  | PascalCase             |
| 函数 / 变量 | camelCase / snake_case |
| 常量      | kCamelCase / ALL_CAPS  |
| 私有成员    | _member                |

---

### 3. 类命名准则

```cpp
class FileReader;
class HttpClient;
```

规则：

* 名词
* 描述“是什么”

---

### 4. 函数命名准则

```cpp
readFile();
parseConfig();
```

规则：

* 动词 + 宾语
* 描述“做什么”

---

### 5. 变量命名准则

```cpp
int fileSize;
bool isValid;
```

规则：

* 表意清晰
* 避免缩写

---

## 四、header + 命名的协同设计

* 头文件是“接口文档”
* 命名是“接口语言”

> 好的头文件，几乎不需要注释。

---

## 五、工程使用准则

* 接口优先设计头文件
* 命名宁长勿歧义
* 统一风格比个人喜好重要

---

## 六、面试级总结

> header / source 分离决定工程结构，命名规范决定代码寿命，这二者是区分“会写代码”和“会做工程”的关键标志。
